## 1. 背景与动机

### 1.1 当前状况

项目正在将 C++ `tbsrc/` 策略层迁移至 Go（`tbsrc-golang/`）。Go 翻译已完成核心框架和 `PairwiseArbStrategy`，但在将 C++ 继承体系映射为 Go 组合模式时遇到了结构性问题。

### 1.2 Go 翻译中的核心痛点

C++ `ExecutionStrategy` 继承体系包含 **42 个子类**，使用虚函数派发（`SendOrder()`, `ORSCallBack()`, `MDCallBack()` 等）。Go 没有继承，当前翻译采用以下 workaround：

| 问题 | Go 的 workaround | 代价 |
|------|------------------|------|
| **虚函数派发** | `ORSCallbackOverride` 字段 + `ProcessORSDirectly()` 方法避免无限递归 | 手工模拟 vtable，脆弱且难维护 |
| **继承层次** | 拆为 `ExecutionState` + `OrderManager` + `LegManager` 三层组合 | 架构变形，与 C++ 原代码无法 1:1 对照 |
| **ExtraStrategy** | C++ 中既是子类又是组件（`extends ExecutionStrategy` 且被 `PairwiseArbStrategy` 持有），Go 无法表达 | 重命名为 `LegManager`，语义偏离 |
| **protected 成员** | C++ 所有字段 public，子类直接访问；Go 跨包需要大写导出 | 封装语义丢失 |
| **接口不一致** | 产生两套接口 `Strategy` vs `StrategyCallback`，签名冲突 | 代码混乱 |

**核心矛盾**: C++ 42 个策略子类如果都要迁移，每个都需要重做组合模式 workaround，工作量和维护成本随规模线性增长。

### 1.3 Java 迁移的动机

Java 的 OOP 继承体系与 C++ 高度一致：
- `abstract class` 对应 C++ 抽象类
- `extends` 对应 C++ 继承
- `@Override` 对应虚函数重写
- `protected` 语义完全一致
- 支持"既是子类又是组件"的设计模式

**目标**: 将策略层核心（TradeBot）用 Java 翻译，保留 C++ CTP 网关（`md_shm_feeder`, `counter_bridge`），通过 SysV MWMR SHM 互操作。

## 2. 架构方案

### 2.1 整体架构

```
保留 C++ 不动:                          新建 Java:
─────────────                          ──────────
md_shm_feeder (C++)                    trader (Java 22+)
  → [SysV MWMR SHM 0x1001] →            ├─ shm/        (Panama FFI: shmget/shmat/MWMR Queue)
                                         ├─ connector/  (SHM 轮询 + 发单/撤单)
counter_bridge (C++)                     ├─ common/     (CommonClient)
  ← [SysV MWMR SHM 0x2001] ←            ├─ strategy/   (ExecutionStrategy 继承体系)
  → [SysV MWMR SHM 0x3001] →            │   ├─ ExecutionStrategy.java (abstract)
                                         │   ├─ ExtraStrategy.java (extends ES)
CTP 接入完全不变                          │   ├─ PairwiseArbStrategy.java (extends ES)
                                         │   └─ ... (按需迁移更多子类)
                                         ├─ indicator/  (指标框架)
                                         └─ config/     (配置加载)
```

**数据流（与 C++ / Go 完全一致）**:
```
md_shm_feeder → [SysV MWMR SHM 0x1001] → Java trader → [SysV MWMR SHM 0x2001] → counter_bridge
                                                      ← [SysV MWMR SHM 0x3001] ← (回报)
```

### 2.2 与 Go 方案的对比

| 维度 | Go (tbsrc-golang) | Java (proposed) |
|------|-------------------|-----------------|
| C++ 翻译忠实度 | **差** — 组合替代继承，架构变形大 | **优** — 继承体系 1:1 映射 |
| SHM IPC | **优** — 原生 syscall，无 GC | **良** — Panama 可行，需 ZGC |
| 代码维护性 | **中** — ORSCallbackOverride 等 hack | **优** — 标准 OOP 继承 |
| 部署简洁性 | **优** — 单二进制 | **良** — 需 JDK 运行时（可 jlink 打包） |
| 扩展策略子类 | **差** — 每个都要重做组合模式 | **优** — 直接 extends |
| 交叉编译 | **优** — `GOOS=linux` | **良** — `jlink` 或容器 |
| GC 暂停风险 | **低** — GC 暂停 ~1ms | **中** — 需用 ZGC，暂停 <1ms |

### 3.2 回调架构

C++ 使用函数指针 + `OrderIDStrategyMap` 做 ORS 回调路由。Java 可直接映射：

```java
// CommonClient.java
public class CommonClient {
    private Connector connector;
    private Consumer<MarketUpdateNew> mdCallBack;
    private Consumer<ResponseMsg> orsCallBack;
    private Consumer<String> indCallBack;

    // C++: CommonClient::SendINDUpdate() — 370行核心分发
    public void sendINDUpdate(MarketUpdateNew raw) {
        String symbol = raw.getSymbol();
        // 1. 更新 Instrument
        // 2. 更新 Indicators
        // 3. indCallBack → SetTargetValue → SendOrder
        indCallBack.accept(symbol);
        // 4. mdCallBack → Strategy.MDCallBack
        mdCallBack.accept(raw);
    }
}
```

**关键优势**: ORS 回调路由中，`PairwiseArbStrategy.orsCallBack()` 自然覆盖 `ExecutionStrategy.orsCallBack()`，不需要 Go 那样的 `ORSCallbackOverride` hack。

## 4. SHM/IPC 层 Java 实现方案

### 4.1 技术选型

**推荐**: Java 22+ Panama Foreign Function & Memory API（stable since JDK 22）

| 需求 | Go 方案 | Java 方案 (Panama) |
|------|--------|-------------------|
| SysV `shmget/shmat` | `syscall.Syscall()` | `Linker.nativeLinker().downcallHandle()` |
| 原子 fetch_add | `sync/atomic.AddInt64()` | `VarHandle.getAndAdd()` on `MemorySegment` |
| 原子 load/store | `atomic.LoadUint64/StoreUint64` | `VarHandle.getVolatile/setRelease()` |
| struct 内存布局 | `unsafe.Pointer` + 手动 padding | `StructLayout` + `paddingLayout()` |
| 内存拷贝 | `unsafe.Slice` + `copy` | `MemorySegment.copy()` |

### 4.2 SysV SHM 封装

```java
// shm/SysVShm.java
public class SysVShm {
    private static final Linker LINKER = Linker.nativeLinker();
    private static final MethodHandle SHMGET = LINKER.downcallHandle(
        LINKER.defaultLookup().find("shmget").orElseThrow(),
        FunctionDescriptor.of(JAVA_INT, JAVA_INT, JAVA_LONG, JAVA_INT));
    private static final MethodHandle SHMAT = LINKER.downcallHandle(
        LINKER.defaultLookup().find("shmat").orElseThrow(),
        FunctionDescriptor.of(ADDRESS, JAVA_INT, ADDRESS, JAVA_INT));

    public static MemorySegment open(int key, long size) throws Throwable {
        int shmid = (int) SHMGET.invoke(key, size, 0);
        return (MemorySegment) SHMAT.invoke(shmid, MemorySegment.NULL, 0);
    }
}
```

### 4.3 消息结构体布局

三个核心结构体必须与 C++ 二进制兼容：

| 结构体 | 大小 | 来源 | 关键挑战 |
|--------|------|------|---------|
| `MarketUpdateNew` | 816 bytes | `marketupdateNew.h` | 含 `bookElement_t[20] × 2`（640 bytes），总体最大 |
| `RequestMsg` | 256 bytes | `orderresponse.h` | `__attribute__((aligned(64)))`，QueueElem 为 320 bytes |
| `ResponseMsg` | 176 bytes | `orderresponse.h` | 3 处显式 padding |

**RequestMsg 对齐问题**: C++ `__attribute__((aligned(64)))` 使 `sizeof(RequestMsg) == 256`，QueueElem 为 320 bytes（非 264）。Java 必须硬编码 `REQ_QUEUE_ELEM_SIZE = 320`。

**Panama StructLayout 定义示例**:

```java
// shm/Types.java

    JAVA_INT.withName("pxType"),                         // offset 108
    JAVA_INT.withName("posDirection"),                    // offset 112
    JAVA_INT.withName("orderID"),                        // offset 116 (uint32 → int)
    JAVA_INT.withName("token"),                          // offset 120
    JAVA_INT.withName("quantity"),                        // offset 124
    JAVA_INT.withName("quantityFilled"),                  // offset 128
    JAVA_INT.withName("disclosedQnty"),                   // offset 132
    JAVA_DOUBLE.withName("price"),                        // offset 136
    JAVA_LONG.withName("timeStamp"),                      // offset 144
    MemoryLayout.sequenceLayout(11, JAVA_BYTE).withName("accountID"),  // offset 152
    JAVA_BYTE.withName("transactionType"),                // offset 163
    JAVA_BYTE.withName("exchangeType"),                   // offset 164

    MemoryLayout.sequenceLayout(32, JAVA_BYTE).withName("product"),    // offset 185
    MemoryLayout.paddingLayout(3),                        // offset 217: 对齐 strategyID

```

### 4.4 MWMR Queue

```java
// shm/MWMRQueue.java
public class MWMRQueue<T> {
    private final MemorySegment base;     // SHM 段
    private final long elemSize;          // sizeof(T) + 8 (seqNo)
    private final long size;              // 2 的幂
    private final long mask;              // size - 1
    private long localTail;               // 本地读指针

    private static final VarHandle HEAD_VH =
        JAVA_LONG.varHandle().withInvokeExactBehavior();

    // 写入（原子 fetch_add 占位 → 拷贝数据 → 发布 seqNo）
    public void enqueue(MemorySegment value) {
        long myHead = (long) HEAD_VH.getAndAdd(base, 0L, 1L);
        long offset = 8 + (myHead & mask) * elemSize;
        MemorySegment.copy(value, 0, base, offset, elemSize - 8);
        VarHandle.storeStoreFence();
        HEAD_VH.setRelease(base, offset + elemSize - 8, myHead);  // seqNo
    }

    // 读取（检查 seqNo → 拷贝数据 → 推进 localTail）
    public boolean dequeue(MemorySegment out) {
        long offset = 8 + (localTail & mask) * elemSize;
        long seqNo = (long) HEAD_VH.getAcquire(base, offset + elemSize - 8);
        if (seqNo < localTail) return false;
        MemorySegment.copy(base, offset, out, 0, elemSize - 8);
        localTail = seqNo + 1;
        return true;
    }
}
```

## 5. 性能评估

### 5.1 延迟指标

| 指标 | C++ | Go (当前) | Java (预期) | 备注 |
|------|-----|----------|------------|------|
| SHM 读写 | < 1μs | < 1μs | < 1μs | Panama/Unsafe 直接内存访问 |
| 策略计算 | < 10ms | < 10ms | < 10ms | 纯计算，JIT 优化后接近 C++ |
| GC 暂停 | 无 | ~1ms (罕见) | **<1ms (ZGC)** | 必须使用 ZGC |
| 端到端延迟 | < 5ms | < 20ms | < 20ms | 满足要求 |
| JIT 预热 | 无 | 无 | ~5-10s | 首次启动时策略响应可能偏慢 |

### 5.2 GC 策略

**强制要求**: 使用 ZGC（`-XX:+UseZGC`）

- ZGC 暂停时间: < 1ms（与 Go GC 相当）
- SHM 数据全部在堆外（Panama `MemorySegment`），不受 GC 影响
- 策略内部对象可正常使用堆内存

**JVM 启动参数**:
```bash
java -XX:+UseZGC -XX:+ZGenerational \
     -Xms256m -Xmx512m \
     --enable-native-access=ALL-UNNAMED \
     -jar trader.jar -config config/trader.92201.yaml
```

### 5.3 JIT 预热缓解

- 策略启动后前 5-10 秒为预热期，JIT 编译热点方法
- 可选: 使用 AOT 编译（GraalVM native-image）消除预热，但会失去 JIT 优化
- 实际影响低: 系统启动时通常有初始化窗口

## 8. 风险分析

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| **GC 暂停影响交易** | 高 | ZGC + 堆外内存（Panama MemorySegment）+ 预热 |
| **Panama API 学习曲线** | 中 | JDK 22 已 stable；文档充足；SHM 层代码量小 |
| **SHM struct 对齐错误** | 高 | 从 Go types_test.go 移植全部 offset 断言 |
| **JDK 22 运行时依赖** | 低 | jlink 打包自包含运行时；CentOS 7+ 支持 |
| **Go 已有投入** | 中 | SHM 层/配置层逻辑可参考；strategy 层重写但更简洁 |
| **JIT 预热期** | 低 | 启动初始化窗口足够；可选 GraalVM AOT |
| **多线程并发** | 低 | 与 Go 一致：粗粒度锁 / 单线程策略（C++ 原本单线程） |

## 10. 结论与建议

### 推荐方案

**采用 Java 翻译策略层核心**，理由：

1. **继承体系 1:1 映射** — 42 个 C++ 策略子类可直接 `extends`，无需组合 workaround
2. **虚函数原生支持** — `@Override` 替代 Go 的 `ORSCallbackOverride` hack
3. **代码可维护性** — 与 C++ 原代码对照审查无架构偏差
4. **扩展性** — 新增策略子类只需 `extends ExecutionStrategy`，零额外模式成本

### 前提条件

- JDK 22+（Panama FFI stable）
- ZGC 作为 GC 策略
- Phase 1 的 offset check 必须 100% 通过后才开始策略翻译

### 与 Go 代码的共存

- Go `tbsrc-golang/` 暂时保留不删除
- Java 和 Go 可以共享同一套 SHM（相同的 key + 相同的 MWMR 协议）
- 逐步验证后切换
